最近花了點時間把幾種「互動網頁」的效果拆開來研究，發現一件有趣的事：它們表面上看起來很像（都是滑一滑、捲一捲、畫面就動），底下的技術棧卻天差地遠。有的是 Three.js、有的刻意不用 Three.js、有的乾脆在瀏覽器裡跑起流體模擬。

這篇是實作筆記，挑三個做法各講一遍原理跟坑。共通的心得先講在前面：**互動效果最會騙人**——螢幕上「看起來會動」跟「你確定它真的照你想的方式在動」是兩回事，所以每一種我都逼自己做一個能反覆重現的小 demo，不是看一眼覺得對就算了。

## 一、Three.js 的高級感效果：液態鉻與體積光

先看那種很吃視覺的產品 landing page，畫面正中一顆會反光的金屬物件，配上一圈柔和的光暈。這類效果我拆出幾個常用的招式：

- **液態鉻／彩虹膜英雄體**（iridescent chrome）
- **體積光碗**（volumetric light bowl，物件底下那圈光池）
- **漸層關鍵字**、**色散捲動轉場**（chromatic aberration）

拆的過程學到最多的，是**新舊版本的落差**。如果你的環境鎖在 Three.js r128，會發現 `MeshPhysicalMaterial.iridescence` 這個屬性根本不存在——那是後來版本才有的。想在 r128 假出虹彩，得改用 `envMap` 搭 `PMREMGenerator`，再靠 `onBeforeCompile` 注入一小段 fresnel 混色的著色器去疊色。r150+ 一行屬性搞定的事，r128 要繞一圈，這種坑不記下來每次都會重踩。

另一個原則是**能不加 post-processing pass 就不加**。體積光碗不必動用 `UnrealBloomPass`：用 canvas 程序生成一張 radial gradient，貼到一個 additive blending 的平面上，再壓扁半圈 `TorusGeometry` 疊上去，光碗就出來了，載入還快。色散轉場也一樣，先用純 CSS（雙偏移的 `text-shadow`）試水溫，真的需要更狠再上 WebGL 的 `ShaderPass`。

**一個 headless 專屬的坑**：如果你想用無頭瀏覽器截圖來驗證畫面，記得把 renderer 開 `preserveDrawingBuffer: true`。不然截出來會是一張空白——頁面明明在動，你卻以為它壞了。

## 二、看起來像 3D，其實是 GSAP 影格捲動

第二個做法很反直覺。有些產品站的 hero，物件會隨著你捲動而 360 度轉一圈，看起來像即時算的 3D，其實**完全沒有 WebGL**。

原理是：把一段預先渲染好的影格序列（幾百張連號 WEBP，`00000`、`00001`⋯）畫到一張 2D `<canvas>` 上，再用 GSAP 的 ScrollTrigger，在一段被 pin 住、高度拉到很長（例如 `height: 700vh`）的區間裡去 scrub。你捲動的位置 = 要顯示第幾張影格。捲動就等於在播放這段影片。

這條路的取捨很清楚：不用扛 3D 場景、光照、模型的重量，畫面又能做到打光完美、質感一致（因為每一幀都是離線 render 好的）。代價是資源檔大、只能沿著固定軌跡看，沒有真正的自由視角。適合「產品定裝照級別」的 hero，不適合要互動旋轉的場景。

實作上有幾個容易漏的細節：

- **斷點用 orientation 而不是 width**：桌面版跟手機版的差異，用 `matchMedia` 判 `orientation`（直向／橫向）比純看寬度更貼近實際體驗。
- **平滑捲動只在桌面開**：ScrollSmoother 這類效果在觸控裝置上反而卡，我會只在橫向桌面才啟用。
- 宣告式的 UI（FAQ 疊層、選單、側邊購物車）交給輕量的框架管狀態就好，不必為了它引入整套重的東西。

## 三、真的在跑 GPU 流體的墨流し

第三個是我最喜歡的，也跳出了「產品頁」的框——做一個日式**墨流し（suminagashi）**的互動畫布。滑鼠或觸控滴一滴、拖一下，墨會像真的染進和紙那樣暈開。

關鍵字是「真的」。它底下跑的是一個 **GPU stable-fluid solver**（Jos Stam 那套半拉格朗日法）。每一步模擬拆成 8 個 pass：curl → vorticity → divergence → pressure → gradient → advect，用 ping-pong 的 framebuffer 來回寫。WebGL2 走主線，偵測不到就退回 WebGL1 的 half-float。這不是 CSS 動畫、也不是放大的圓形漸層，是實打實在解流體場。

真正讓它「像墨」的是**染色模型**。我用 Beer–Lambert 定律：`paper * exp(-absorb)`，而且四種傳統墨色（墨、紺、朱、松葉）存的不是 RGB，是逐通道的**吸收向量**。朱色之所以是朱色，是因為它吸收綠光和藍光、把紅光留下來。如果偷懶改成 alpha 疊色，出來的東西一點都不像墨——**物理對了，顏色才會對**，這是它跟一般 canvas 玩具的分水嶺。和紙也是程序生成的（纖維 + 顆粒 + 暗角），閒置幾秒還會自動開始畫，讓頁面即使沒人碰也有東西看。

**跟捲動耦合**是它跟前面兩個做法接起來的地方：把整塊墨流し畫布設成 `position: fixed; pointer-events: none; z-index: 0` 的固定背景，上面疊會捲動的內容。然後用一個 passive、rAF-throttled 的 scroll 監聽器，依照捲動的 delta 往流體場注入一發 velocity splat——於是**捲動會攪動墨水**。背景不是一張貼圖，是你往下滑的時候真的被你推動的流體。這種「內容在動、背景也跟著被你影響」的層次，比純視差有意思得多。

## 收尾：怎麼確定它「真的在動」

三個做法用的技術差這麼多，但我驗收的方式其實一樣：做一個最小可重現的 demo，然後**用可以量測的行為去確認**，而不是肉眼掃一眼就簽收。

- 鉻金屬 hero：headless 截圖不能是空白（所以才有前面那個 `preserveDrawingBuffer` 的坑）。
- 影格捲動：捲到某個位置，畫面上就該是對應的那一幀。
- 墨流し：滴一滴，紙就該變暗；按「洗い流す」，畫面就該淡回乾淨；閒置時 autoplay 得看得到變化；控制列在手機直向（例如 390×844）不能溢出畫面。

把判準釘在「可執行、可重現」上，我才敢說一個互動效果真的能用，而不是 demo 一關掉就露餡。互動網頁做久了會發現，難的通常不是把效果做出來，是**確定它在各種裝置、各種狀態下都還是你要的那個樣子**。這三個小東西，其實都是在練這件事。
